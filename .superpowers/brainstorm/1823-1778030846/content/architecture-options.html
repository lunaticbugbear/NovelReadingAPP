Choose the foundation for the novel reader webapp

Since we are not converting Mihon directly, the main choice is how much backend we want in the first version.

A

Browser-only PWA

Everything runs in the browser. URLs are configured locally, chapters are fetched directly where possible, and reading state is stored in IndexedDB.

Pros

  • Fastest to build
  • No server to deploy
  • Works offline after caching

Cons

  • Many sites block browser fetching with CORS
  • Harder to normalize messy source pages
B

Webapp + source-fetch backend

A frontend for library/reader UI plus a backend API that fetches embedded source URLs, parses chapters, and shields the browser from CORS issues.

Pros

  • Most reliable for arbitrary URLs
  • Cleaner parsing pipeline
  • Easier syncing later

Cons

  • Requires server hosting
  • Slightly more work up front
C

Static catalog first, backend later

Start with novels and chapters embedded as config/JSON or pre-imported content, then add live URL fetching after the reader experience is solid.

Pros

  • Lowest risk for first UI prototype
  • No scraper complexity at first
  • Great for validating reader UX

Cons

  • Does not solve live sources immediately
  • More migration work later
